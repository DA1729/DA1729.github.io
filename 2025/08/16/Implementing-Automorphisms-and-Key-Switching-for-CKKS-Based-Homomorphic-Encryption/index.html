Automorphisms and Key Switching in LWE based Cryptosystems

#Cryptography #Post Quantum Cryptography #Fully Homomorphic Encryption #Abstract Algebra

In my previous blogs on LWE cryptanalysis and Ring-LWE foundations, I covered the basic mathematical structures underlying modern FHE schemes. If you haven’t read those, especially the Ring-LWE and CKKS mathematical foundations post, I’d recommend checking that out first since I’ll be building directly on those concepts here.

Now, I want to tackle a super specialized but crucial topic that honestly has embarrassingly little literature online - automorphisms and key switching in Ring-LWE based systems. This is gonna get pretty heavy on Galois theory and some advanced algebraic techniques, but we’ll take it step by step.

As I covered in my Ring-LWE foundations post, almost all modern FHE schemes are built around the Ring-LWE problem over cyclotomic rings. We can perform addition and multiplication between ciphertexts homomorphically, but there’s another crucial operation that’s absolutely essential for practical applications - ciphertext rotation.

Now, rotation is used in places like shifting an image feature map in a CNN, or cycling through records in an encrypted data. But, rotating a ciphertext should feel counter-intuitive cuz encryption is designed to make data look random and to be able to rotate it should destroy the meaning right? Well, no. Cuz the way we encrypt the data using LWE can be viewed as the process of adding noise to the data, so when we rotate it, the noise kind of “shuffles” and the ciphertext would look very different than it looked before and we won’t be able to figure out any patterns.

For this very purpose of rotation, I am going to use a very elegant concept from algebra: Automorphisms. Automorphisms, we are going to see it in more details further, but for now, they are “structure-preserving” transformations that allow us to rotate the underlying data within its encrypted shell. But, this comes with an additional cost, i.e., after performing an automophism over the LWE-ciphertext, the key under which it was originally encrypted with, also gets “automorphed” or also gets rotated. Now, we can just ask the client to decrypt it using the rotated key, but that’s not practical cuz the client would have to store multiple keys and also know which key corresponds to which rotation. That’s where Key Switching comes in - it’s a technique that allows us to transform a ciphertext encrypted under one key to be decryptable under a different key, without ever revealing the plaintext.

Galois Theory and Automorphism Groups

Alright, so to really understand automorphisms in the context of Ring-LWE, we need to dive into some Galois theory. Don’t worry if you’re not super familiar with this stuff - I’ll build it up from the ground.

The Galois Group of Cyclotomic Fields

Recall from my Ring-LWE foundations post that we work in the ring R=Z[X]/(Xn+1)R = \mathbb{Z}[X]/(X^n + 1)R=Z[X]/(Xn+1) where nnn is a power of 2. The polynomial Xn+1X^n + 1Xn+1 is the 2n2n2n-th cyclotomic polynomial Φ2n(X)\Phi_{2n}(X)Φ2n​(X).

The splitting field of Φ2n(X)\Phi_{2n}(X)Φ2n​(X) over Q\mathbb{Q}Q is the cyclotomic field Q(ζ2n)\mathbb{Q}(\zeta_{2n})Q(ζ2n​), where ζ2n=e2πi/2n\zeta_{2n} = e^{2\pi i / 2n}ζ2n​=e2πi/2n is a primitive 2n2n2n-th root of unity.

The Galois group Gal(Q(ζ2n)/Q)\text{Gal}(\mathbb{Q}(\zeta_{2n})/\mathbb{Q})Gal(Q(ζ2n​)/Q) is isomorphic to the group of units (Z/2nZ)∗(\mathbb{Z}/2n\mathbb{Z})^*(Z/2nZ)∗. Each element k∈(Z/2nZ)∗k \in (\mathbb{Z}/2n\mathbb{Z})^*k∈(Z/2nZ)∗ corresponds to the automorphism σk\sigma_kσk​ defined by:

σk:ζ2n↦ζ2nk\sigma_k: \zeta_{2n} \mapsto \zeta_{2n}^kσk​:ζ2n​↦ζ2nk​

Since nnn is a power of 2, we have (Z/2nZ)∗≅Z2×Zn/2(\mathbb{Z}/2n\mathbb{Z})^* \cong \mathbb{Z}_2 \times \mathbb{Z}_{n/2}(Z/2nZ)∗≅Z2​×Zn/2​ when n≥4n \geq 4n≥4.

Structure of the Automorphism Group

For n=2dn = 2^dn=2d with d≥2d \geq 2d≥2, the group (Z/2nZ)∗(\mathbb{Z}/2n\mathbb{Z})^*(Z/2nZ)∗ has order ϕ(2n)=n\phi(2n) = nϕ(2n)=n and can be generated by:

  • g1=−1 mod 2ng_1 = -1 \bmod 2ng1​=−1mod2n (order 2): corresponds to complex conjugation
  • g2=5 mod 2ng_2 = 5 \bmod 2ng2​=5mod2n (order n/2n/2n/2): corresponds to "rotations"

Every automorphism can be written uniquely as σk\sigma_kσk​ where k=(−1)b0⋅5b1k = (-1)^{b_0} \cdot 5^{b_1}k=(−1)b0​⋅5b1​ for b0∈{0,1}b_0 \in \{0,1\}b0​∈{0,1} and b1∈{0,1,…,n/2−1}b_1 \in \{0, 1, \ldots, n/2-1\}b1​∈{0,1,…,n/2−1}.

When we extend these automorphisms to act on polynomials in Z[X]/(Xn+1)\mathbb{Z}[X]/(X^n + 1)Z[X]/(Xn+1), we get:

σk:f(X)↦f(Xk) mod (Xn+1)\sigma_k: f(X) \mapsto f(X^k) \bmod (X^n + 1)σk​:f(X)↦f(Xk)mod(Xn+1)

  • σk(f+g)=σk(f)+σk(g)\sigma_k(f + g) = \sigma_k(f) + \sigma_k(g)σk​(f+g)=σk​(f)+σk​(g)

To compute σk(f(X))\sigma_k(f(X))σk​(f(X)) where f(X)=∑i=0n−1fiXif(X) = \sum_{i=0}^{n-1} f_i X^if(X)=∑i=0n−1​fi​Xi, we need to evaluate:

Automorphisms in Cyclotomic Rings

Now let’s see how these automorphisms actually work in practice and why they’re so useful for ciphertext rotations.

Action on CKKS Slots

Remember from my Ring-LWE foundations post that CKKS uses the Chinese Remainder Theorem to pack n/2n/2n/2 complex numbers into the n/2n/2n/2 slots corresponding to roots {ζ2n,ζ2n3,…,ζ2n2n−1}\{\zeta_{2n}, \zeta_{2n}^3, \ldots, \zeta_{2n}^{2n-1}\}{ζ2n​,ζ2n3​,…,ζ2n2n−1​}.

When we apply automorphism σk\sigma_kσk​ to a polynomial, it permutes these slots according to the map:

slot i↦slot (ki mod 2n)\text{slot } i \mapsto \text{slot } (ki \bmod 2n)slot i↦slot (kimod2n)

But since we only have nnn actual roots (the odd powers), the effective permutation on the n/2n/2n/2 conjugate pairs is more complex.

Generators and Rotations

The most practically important automorphisms are:

  1. Complex Conjugation (k=−1k = -1k=−1): Maps ζ2n2i+1↦ζ2n2n−1−(2i+1)=ζ2n2i+1‾\zeta_{2n}^{2i+1} \mapsto \zeta_{2n}^{2n-1-(2i+1)} = \overline{\zeta_{2n}^{2i+1}}ζ2n2i+1​↦ζ2n2n−1−(2i+1)​=ζ2n2i+1​​
  2. Rotations (k=5jk = 5^jk=5j): Cyclically shift the slots

For rotations, the key insight is that k=5jk = 5^jk=5j generates a cyclic subgroup of order n/2n/2n/2 in (Z/2nZ)∗(\mathbb{Z}/2n\mathbb{Z})^*(Z/2nZ)∗. This gives us exactly the rotations we want for SIMD operations.

Frobenius Endomorphisms

There’s a beautiful connection to Frobenius endomorphisms here. In characteristic ppp, the Frobenius map x↦xpx \mapsto x^px↦xp is always a ring endomorphism. Our automorphisms σk\sigma_kσk​ can be viewed as “multiplicative Frobenius” maps.

When working modulo q=∏piq = \prod p_iq=∏pi​, we can apply the Chinese Remainder Theorem again:

Zq[X]/(Xn+1)≅∏iZpi[X]/(Xn+1)\mathbb{Z}_q[X]/(X^n + 1) \cong \prod_i \mathbb{Z}_{p_i}[X]/(X^n + 1)Zq​[X]/(Xn+1)≅∏i​Zpi​​[X]/(Xn+1)

The automorphism σk\sigma_kσk​ acts componentwise, and on each component Zpi[X]/(Xn+1)\mathbb{Z}_{p_i}[X]/(X^n + 1)Zpi​​[X]/(Xn+1), it’s a genuine ring automorphism.

Orbit-Stabilizer Analysis

The automorphism group acts on the set of n/2n/2n/2 slot positions. For practical CKKS implementations, we usually care about:

  • Full rotations: Generated by σ5\sigma_5σ5​, giving all n/2n/2n/2 rotations
  • Power-of-2 rotations: σ52i\sigma_{5^{2^i}}σ52i​ for efficient “butterfly” patterns
  • Conjugation: σ−1\sigma_{-1}σ−1​ for complex conjugation

The orbit of any slot under the full automorphism group has size n/2n/2n/2 (assuming n≥4n \geq 4n≥4), meaning every slot can be rotated to every other slot.

The Key Switching Problem

Here’s the fundamental issue that makes automorphisms tricky in practice: when we apply an automorphism to a Ring-LWE ciphertext (a(x),b(x))(a(x), b(x))(a(x),b(x)), we get:

(σk(a(x)),σk(b(x)))(\sigma_k(a(x)), \sigma_k(b(x)))(σk​(a(x)),σk​(b(x)))

But now, this ciphertext is encrypted under the key σk(s(x))\sigma_k(s(x))σk​(s(x)), not the original key s(x)s(x)s(x)!

The Mathematical Problem

Starting with a Ring-LWE sample: b(x)=a(x)⋅s(x)+e(x)+Δ⋅m(x)b(x) = a(x) \cdot s(x) + e(x) + \Delta \cdot m(x)b(x)=a(x)⋅s(x)+e(x)+Δ⋅m(x)

After applying automorphism σk\sigma_kσk​:

σk(b(x))=σk(a(x))⋅σk(s(x))+σk(e(x))+Δ⋅σk(m(x))\sigma_k(b(x)) = \sigma_k(a(x)) \cdot \sigma_k(s(x)) + \sigma_k(e(x)) + \Delta \cdot \sigma_k(m(x))σk​(b(x))=σk​(a(x))⋅σk​(s(x))+σk​(e(x))+Δ⋅σk​(m(x))

This follows because automorphisms are ring homomorphisms, so they preserve multiplication and addition.

The Problem: Our rotated ciphertext (σk(a(x)),σk(b(x)))(\sigma_k(a(x)), \sigma_k(b(x)))(σk​(a(x)),σk​(b(x))) now encrypts the rotated message σk(m(x))\sigma_k(m(x))σk​(m(x)) (which is what we want!) but under the rotated key σk(s(x))\sigma_k(s(x))σk​(s(x)) (which we don’t want).

The client only possesses the original secret key s(x)s(x)s(x), not the entire orbit {σk(s(x)):k∈(Z/2nZ)∗}\{\sigma_k(s(x)) : k \in (\mathbb{Z}/2n\mathbb{Z})^*\}{σk​(s(x)):k∈(Z/2nZ)∗}.

Formalizing Key Switching

Definition: Given ciphertexts (a,b)(a, b)(a,b) encrypting message mmm under key s′s's′, key switching produces ciphertext (a′,b′)(a', b')(a′,b′) encrypting the same message mmm under key sss.

The challenge is doing this without ever learning mmm, sss, or s′s's′ in the clear.

Security Requirement: The key switching operation should be semantically secure - an adversary seeing the input ciphertext, output ciphertext, and public switching key learns nothing about the message or secret keys.

The General Key Switching Framework

Key switching works by publishing auxiliary information called switching keys that encode the relationship between sss and s′s's′ in a way that allows transforming ciphertexts without revealing the keys themselves.

The idea is to encrypt s′s's′ under sss, then use this to “substitute” s′s's′ for sss in the ciphertext equation.

Formal Security Analysis of Key Switching

Now let’s dive deep into the mathematical foundations and security analysis of key switching. This is where things get really rigorous.

Digit Decomposition and Base-B Representation

The naive approach to key switching has a serious problem: noise explosion. If we directly encrypt each coefficient of s′s's′, the noise grows linearly with the coefficient magnitude, which can be as large as qqq.

The solution is digit decomposition. We choose a base BBB (typically B=2ℓB = 2^{\ell}B=2ℓ for some small ℓ\ellℓ) and decompose elements a∈Zqa \in \mathbb{Z}_qa∈Zq​ as:

a=∑i=0L−1aiBi where ai∈{0,1,…,B−1} and L=⌈log⁡Bq⌉a = \sum_{i=0}^{L-1} a_i B^i \text{ where } a_i \in \{0, 1, \ldots, B-1\} \text{ and } L = \lceil \log_B q \rceila=∑i=0L−1​ai​Bi where ai​∈{0,1,…,B−1} and L=⌈logB​q⌉

This is the standard base-B representation, but we need to be careful about the range of coefficients.

Specialized Key Switching for Automorphisms

For automorphism-based rotations, we have a special structure. Starting with ciphertext (a,b)(a, b)(a,b) under key sss, we apply σk\sigma_kσk​ to get (σk(a),σk(b))(\sigma_k(a), \sigma_k(b))(σk​(a),σk​(b)) under key σk(s)\sigma_k(s)σk​(s).

  1. Compute b∗=σk(b)−σk(a)⋅sb^* = \sigma_k(b) - \sigma_k(a) \cdot sb∗=σk​(b)−σk​(a)⋅s
  2. Apply standard key switching to b∗b^*b∗ viewed as encrypting some message under key σk(s)−s\sigma_k(s) - sσk​(s)−s

Advanced: Bootstrapping Key Switching

In the context of bootstrapping (refreshing ciphertext noise), key switching becomes even more complex. The bootstrapping procedure requires switching from a “small” key with coefficients in {0,1}\{0, 1\}{0,1} to the “large” key used for homomorphic operations.

This requires gadget-based key switching where the switching key has a tensor product structure, enabling more efficient noise control.

The analysis involves spectral norms of the gadget matrix and requires more sophisticated techniques from the geometry of numbers.

CKKS Scheme Overview

The CKKS (Cheon-Kim-Kim-Song) scheme is a particularly elegant homomorphic encryption scheme that supports approximate arithmetic on complex numbers. It’s built on Ring-LWE and allows us to pack multiple values into a single ciphertext using the Chinese Remainder Theorem.

In CKKS, a plaintext is encoded as a polynomial where the coefficients encode the real and imaginary parts of complex numbers. When we perform automorphisms on CKKS ciphertexts, we’re essentially rotating these packed values.

For example, if we have n/2n/2n/2 slots packed with complex numbers z0,z1,…,zn/2−1z_0, z_1, \ldots, z_{n/2-1}z0​,z1​,…,zn/2−1​, then applying automorphism σk\sigma_kσk​ gives us a rotation of these slots.

Security Considerations

Now, implementing this stuff comes with some serious security considerations that you absolutely cannot ignore if you’re planning to use this in any real application:

  1. Parameter Selection: The values of nnn, qqq, and noise distribution need to be chosen carefully to resist lattice attacks. What I’ve used here are toy parameters for demonstration.

  2. Key Switching Noise Growth: Each key switching operation adds noise to the ciphertext. Too many operations can make the noise so large that decryption fails.

  3. Switching Key Size: The switching keys can be quite large - for each automorphism you want to support, you need O(n)O(n)O(n) Ring-LWE samples.

  4. Side-Channel Attacks: The implementation I’ve shown doesn’t protect against timing attacks or other side-channel attacks. Production implementations need constant-time operations.

  5. Random Number Generation: Using a proper cryptographically secure random number generator is crucial. The std::random_device I’ve used might not be sufficient for real applications.

Real-World Applications

This combination of automorphisms and key switching is super useful in practice:

  • Private Neural Network Inference: Rotating feature maps in encrypted CNNs
  • Encrypted Database Queries: Shifting encrypted records to find matches
  • Secure Multi-Party Computation: Redistributing encrypted data among parties
  • Privacy-Preserving Machine Learning: Performing encrypted matrix operations

The math might seem daunting at first, but once you get the hang of it, it’s actually quite elegant. The ring structure gives us this beautiful algebraic machinery that lets us perform complex operations on encrypted data.
